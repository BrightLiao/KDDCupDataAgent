类与职责

类 / 模块位置职责

Prompt 三段结构

  • System：REACT_SYSTEM_PROMPT — 7 条硬约束（探索后回答 / 仅依赖工具观察 / 必须调 answer / answer 是表格 / 单 JSON 对象 / 用 ```json 围栏 / 围栏外不能有文字）
  • User (task)：question + 数据目录列表 + AnswerTable schema 提示
  • Assistant + tool observation 交替追加：每步 LLM 输出原文进 messages，对应 observation 作为下一轮 user 消息

System Prompt 关键约束（节选自 prompt.py）

1. Use tools to inspect the available context before answering.
2. Base your answer only on information you can observe through the provided tools.
3. The task is complete only when you call the `answer` tool.
4. The `answer` tool must receive a table with `columns` and `rows`.
5. Always return exactly one JSON object with keys `thought`, `action`, `action_input`.
6. Always wrap that JSON object in exactly one fenced code block ```json ... ```.
7. Do not output any text before or after the fenced JSON block.

Action 协议

```json
{
  "thought": "...",
  "action": "<tool_name>",
  "action_input": { ... }
}
```
  • action ∈ {list_context, read_csv, read_json, read_doc, inspect_sqlite_schema, execute_context_sql, execute_python, answer}
  • action_input 为该 tool 的入参 dict
  • 解析失败 → 注入 __error__ observation："Failed to parse action: ..."；下一轮继续
  • 答案出口仅有 answer：input 必须包含 {columns: [str], rows: [[...]]}，由 Runner 写成 prediction.csv
⚠️ 已观察到的失败模式（实跑数据）
JSON 解析失败步占比 33.7%（163 / 483 步）。 当 LLM 把 Python 代码塞进 action_input.code 时，三引号字符串 / 反斜杠转义 / 嵌入双引号经常破坏 JSON 结构，触发 __error__ 状态。模型温度=0 时陷入"重复同样错误"循环，耗尽 max_steps。把 max_steps 从 16 拉到 30、超时从 600s 拉到 1200s 重试一遍后仍有 10 个任务无法 submit：3 个 max_steps（task_19 / 250 / 396），7 个 1200s 超时（task_257 / 269 / 344 / 352 / 379 / 418 / 89）—— 证明这是协议层 bug，加大预算救不回。
  • 这是 ReAct + 严格 JSON action 的已知通病，CodeAct (arXiv:2402.01030) §3 就是为解决这个
  • 修复路径：(a) 把 action 改成原生 markdown 代码块而非 JSON 字段；(b) 严格校验失败时反馈具体的 char 位置 + 修复指令；(c) 温度 > 0 引入随机性逃出循环

⑥ Tools 设计

注册机制

ToolRegistry 维护 {name → ToolHandler} 映射；每个 handler 暴露 name / description / input_schema (JSON Schema) / execute(input) → ToolResult 四要素。create_default_tool_registry() 在 tools/registry.py 一次性注册 8 个内置 handler；新增 tool 只需追加 handler 实例。

八个 Tool 详细规格

Tool入参返回用途 / 约束
list_context—{ files: [path], total }列出 task context/ 下文件树（max_depth=4），第一步用来探索数据资产
read_csv{ path, max_rows=20 }{ columns, rows, dtype_summary }仅做预览，不加载全表；用于判断 schema / 数据分布
read_json{ path, max_chars=4000 }{ text, truncated }读 task.json / 元数据；超过 4K 字符自动截断
read_doc{ path, max_chars=4000 }{ text, truncated }读 *.md / *.txt 文档资产（hard 任务里的散文型 schema 描述）
inspect_sqlite_schema{ path }{ tables: [{name, columns, sample_rows}] }列出 .sqlite/.db 全部表 + 列 + 3 行采样
execute_context_sql{ path, sql, limit=200 }{ columns, rows }只读模式（pragma query_only）；自动 LIMIT 200；medium 多表 JOIN 主力
execute_python{ code, timeout=30 }{ stdout, stderr, succeeded }子进程沙箱，cwd=task context/；30s 超时；通用兜底
answer ◉ 终止{ columns: [str], rows: [[...]] }—唯一终止 tool：写 prediction.csv 后跳出循环

沙箱与安全

  • execute_python：用 subprocess.run 启动新 Python 解释器，cwd 限定到该 task 的 context/，不共享父进程命名空间；30s timeout；stdout/stderr 长度截断防上下文炸开
  • execute_context_sql：以 file:...?mode=ro 打开 sqlite + PRAGMA query_only=1，结构上禁止 INSERT/UPDATE/DELETE
  • filesystem read_*：路径必须在 context/ 子树下，禁止 .. 穿越

⑨ 核心结论 + 改进路径

  • 真实基线 micro Score = 0.5153 / macro Score = 0.4022（提交率 80%，但只有 50% 左右是对的）—— 后续每次迭代的对照分母。
  • 按难度：Easy 0.5789 · Medium 0.5217 · Hard 0.5083 · Extreme 0.0000 —— Easy/Medium 各有 ~5 个零分（提交了但列匹配不上），Extreme 完全失守。
  • JSON 解析失败是头号 bottleneck。__error__ 占 33.7%，且 把 max_steps 从 16 拉到 30、超时从 600s 拉到 1200s 后仍有 10 个任务无法 submit（task_19/250/396 max_steps，task_257/269/344/352/379/418/89 timeout）—— 协议层 bug，加大预算救不回。
  • Score < 提交率：80% 提交、51% 得分意味着 ~14 个任务"答对了交了但列不对"（task_25 事件不匹配、task_163 漏聚合、task_180 多余列、task_259 列数过多、task_420 99.93 vs 100.0 浮点对齐）—— 输出归一化和列裁剪有立等可见的 ROI。
  • 下一步建议（按 ROI 排序）：
    1. ReAct → CodeAct 范式（arXiv:2402.01030）：直接出 ```python 代码块，绕开 JSON action_input 转义陷阱 —— 消灭 33.7% 错误步 + 救回 10 个未提交
    2. 输出归一化器：submit 前强制 round(2) / ISO 日期 / 空值统一 —— 直接救回 task_420 类零分
    3. Schema 预览注入 prompt：启动时跑一次 head -3 + dtypes，省 4–6 步预算
    4. 留 15 个 holdout 做反过拟合对照，所有 prompt/threshold/few-shot 仅在 35 个 train 上调
    5. 跑 DABench 公开版 257 任务做外部验证集，再决定是否上 Text2SQL 子工具
