# Baseline 改进路线图

> 当前 baseline：micro Score 0.5153 / macro 0.4022 / 提交率 80% / 错误步占比 33.7%（[reports/baseline_architecture.html](../reports/baseline_architecture.html)）。
>
> **方法论硬约束**：本文档不允许出现"针对 task_X 加一条规则"。每条改进必须是**类级别机制**——有名字、有适用条件、能在没见过的同类问题上复用。说不出"它处理的是哪一类"就不做。
>
> **验证用两层**：内层用全部 50 demo 题看过程指标（parse 失败率、`__error__` 率、平均步数、提交率）；外层每个 Phase 收尾时跑 DABench 外部样本（arXiv:2407.15838）。**不在 50 题里再切盲测**——50 题样本量太小，留 10 题做盲测既损失开发信号又给假阳性，外部数据集才是真正的"未见过同类"。

## 改进按层组织（4 层）

每层处理不同**类**的失败，建议从 L1 顺序往下做：

| 层 | 处理的问题类 | 机制 | 论文锚点 |
| --- | --- | --- | --- |
| L1 协议层 | 工具与 LLM 之间的翻译协议 | **M1**：CodeAct 代码块替换 JSON action | arXiv:2402.01030 |
| L2 上下文/知识层 | LLM 看到的信息完备性 | **M2**：全量 schema 预注入 + knowledge.md 全文 + REPL 预加载 | — |
| L3 答案控制层 | 提交内容的形状 + 表示是否对齐 scorer | **M3**：归一化器 + 形态约束（合并版） | — |
| L4 架构层 | 控制结构是否避免漂移、能否做长程规划 | **M4**：Plan-Executor with Replan（按难度路由） | arXiv:2305.04091 / 2312.04511 / 2509.21825 |

---

## L1 协议层

### M1. JSON action → CodeAct 代码块

**问题类**：结构化输出协议陷阱。LLM 把 Python 代码塞进 JSON 字符串字段时，三引号、换行、反斜杠的双重转义错一字符全步作废（当前 33.7% 错误步）。

**类边界**：所有需要 LLM 输出可执行/结构化负载的场景，**不仅限于 Python 代码**——同样适用于 SQL、JSON Schema 输出、长字符串模板。

**机制**：参考 CodeAct（arXiv:2402.01030 §3）。删掉 `agents/react.py` 的 `_strip_json_fence` + `_load_single_json_object`，换成 ` ```python ... ``` ` 代码块解析。Tool 接口由 LLM 直接调函数（`read_file()` / `run_sql()` / `submit()`）替代 `{"action": ..., "action_input": ...}` 协议。

**适用条件**：当 LLM 需要在 action_input 里携带"含特殊字符的长 payload"时启用。短结构化参数（如 `{"path": "foo.csv"}`）保留 JSON 也行——但为统一接口建议全切。

**落点**：

- `agents/codeact.py`（新）
- `agents/prompt.py` 增 `CODEACT_SYSTEM_PROMPT`
- `tools/python_exec.py` 改持久 REPL（同 task 跨步共享变量，是 L2 / L4 的前置条件）
- `cli.py` 加 `--agent {react,codeact}`

**预期**：消灭 33.7% 错误步中的协议解析失败部分；救回 task_11 / 19 / 250 / 396 等"反复 `__error__`"任务。

## L2 上下文/知识层

### M2. 全量 Schema 预注入 + knowledge 全文 + REPL 预加载

**问题类**：探索预算浪费 + 截断导致的 schema 误判 + 领域知识被冷落。Agent 前 4–6 步在反复 `list_dir` / `read_file` 看数据长啥样，挤占解题预算（max_steps=16 时占 25–37%）；现有 `read_file` 工具对大文件做字节截断，LLM 看到的只是前几条记录，对列分布、稀疏度、特殊值（异常码、外键）一无所知，会基于残缺 schema 写错查询；`knowledge.md` 是出题方给的领域必备知识，但 ReAct 模式下不一定读到。

**类边界**：所有 KDD Cup 任务。输入结构是**固定**的（`task.json` + `context/{csv,json,db,doc/}` + `context/knowledge.md`），不需启发式扫描，可硬编码遍历策略。

**机制**：runner 启动时对 `task.context_root` 做**全量结构化扫描——全读不截断，但只把派生摘要返回给 LLM**：

| 文件类型 | 全量动作（runtime 内存） | 注入 prompt 的内容 |
| --- | --- | --- |
| `context/csv/*.csv` | `pd.read_csv` 全表 | 列名 + dtypes + `describe()` + 每列 null 率 + 类别列基数 + 高基数列 top-5 取值 + 日期列 min/max + 行数 |
| `context/json/*.json` | `json.load` 全部 | 顶层结构树（递归 keys + 类型）+ records 数量 + 首条样例 + 嵌套字段路径清单 |
| `context/db/*.db` (SQLite) | 连接 + 列出全部表 | 每表 schema (CREATE TABLE) + 行数 + `SELECT * LIMIT 3` + 主键/外键关系 |
| `context/knowledge.md` / `context/doc/*.md` | 读全文（通常 < 10KB） | **全文原样注入** —— 这是出题方提供的领域知识，必读必用 |
| `context/doc/*.sql` | 读全文 | 全文原样注入（视图/查询模板） |

**关键设计**：

1. **"全读 + 摘要"双轨**：raw payload 不返回 LLM（防 token 爆 + 防 prompt 注入），但全量 DataFrame / dict / sqlite connection **预加载到 CodeAct REPL 命名空间**，命名规则固定（`df_<filename>` / `json_<filename>` / `conn_<dbname>`），LLM 第一步代码就能直接用。
2. **`knowledge.md` 全文必读**：不做 RAG / 摘要，原文塞 system prompt。出题方给的就这点字数，省下来反而失分。
3. **schema 摘要工具留作 LLM 主动调**：起名 `inspect(name)`，给已加载对象再返回更详细的统计（如 `inspect("df_patient")` 返回 value_counts / 相关性矩阵），用于深入探索时不再读文件。

**强耦合 M1**：必须 CodeAct 持久 REPL 才能做 "全读不截断 + 不返回原文"——ReAct 协议下每步独立子进程，预加载的 df 跨步会丢。所以 L1 → L2 顺序不能反。

**适用条件**：所有任务，零启发式。Token 预算上：50 题里最大的 csv 摘要 ~3KB，最大 sqlite ~8KB，knowledge.md 全文均值 ~5KB，总注入 < 30KB（Qwen3.5 256K 上下文 1% 都不到）。

**预期**：

- 每题节省 4–6 步纯探索（消灭 list_dir / read_file 循环）
- 修复 read_file 截断导致的 schema 误判类错误（task_38/64/200 等多 csv 任务里 LLM 经常看不到关键列）
- 间接给 Hard/Extreme 题更多步预算做实际计算
- 同时也是 L4 Plan-Executor 的前置——Planner 没全量 schema 就是盲规划

## L3 答案控制层

### M3. 答案归一化 + 形态约束（合并版）

**问题类**：提交内容到 scorer 那里失分，分两个相邻子类：

- **格式不对齐**：日期 `2024/4` vs `2024-04`、浮点 `0.1` vs `0.10`、空值 `nan` vs `""`、字符串大小写/前后空格——评分用 multiset 严格比对（`src/eval/scorer.py:38`），表示差一点就 0 分
- **形态不对齐**：多输出 1 列、少 1 列、行数远超提示——这些题"看起来 submit 了"但被 scorer 因 `Extra Columns / Predicted Columns` 项重罚

**类边界**：所有提交字段；以及 question 里有明确"几列、几行"指示的题（模糊题如"列出相关信息"跳过形态约束）。

**机制**：在 `submit()` 调用点（CodeAct 命名空间下是 `submit(df)` 函数）做**两重校验 + 一次归一化**：

1. **形态校验（先）**：
   - runner 启动时让 LLM 解析 question，强制输出 `{"expected_columns": [...], "expected_row_count_hint": "top-5" / "all matching" / null}` 塞进 state
   - submit 时列数不对 → 拒绝 + 把约束回喂 LLM 让它修；行数提示明确（如 top-5）→ 截断/警告
2. **值归一化（后）**：通过形态校验的 df 走 normalizer：
   - 日期 → ISO 8601（`YYYY-MM-DD`）
   - 数值 → 跟随题目精度提示 `round(N)`，无提示则保留原值
   - 空值 → 统一空字符串（不是 `nan` / `None` / `null`）
   - 字符串 → `strip()`（不强制 `lower()`，因有些题大小写敏感）

**实现锚点**：归一化逻辑与 `src/eval/scorer.py:38` 的 `normalize_value` **完全对齐**——评分器怎么 normalize，提交端就怎么 normalize。两边抽公共模块 `src/eval/normalize.py`，物理上保证不漂移。

**预期**：

- 救回当前"提交了但 Score=0"的 ~14 题（task_25 / 80 / 86 / 163 / 169 / 173 / 180 / 196 / 199 / 200 / 259 / 420 等）
- 把 λ·(Extra/Pred) 扣分项从平均 ~0.1 拉到接近 0
- 对 Recall 项无负面影响（只是更严格地对齐 scorer 期望）

## L4 架构层

### M4. Plan-Executor with Replan（按难度路由）

**问题类**：控制结构性缺陷。Flat ReAct/CodeAct 循环在弱模型 + 长程任务下出现两类失败：

- **死循环漂移**：同一 action + 同一错误反复重试到 `max_steps`（task_19 / 250），3 次失败也不主动换路径
- **缺乏全局视图**：每步只看上一步的观察，没有"我现在执行到 plan 的第几步、距离目标还差什么"的概念，Hard/Extreme 题解题路径长直接超预算

这是控制结构问题，不是协议问题。M1 的 CodeAct 改的是"怎么说"，M4 改的是"怎么思考"。

#### 主流架构对比（在 Qwen3.5-35B-A3B 单一 LLM + CPU-only + 列 multiset 比对约束下）

| 架构 | 控制结构 | 强项 | 弱项 | 我们任务下评估 |
| --- | --- | --- | --- | --- |
| ReAct (arXiv:2210.03629) | 单循环 thought→action→obs | 简单、低延迟 | 弱模型易死循环；无全局规划 | **当前 baseline，已暴露问题**；Easy 够用，Hard/Extreme 不行 |
| CodeAct (arXiv:2402.01030) | 同 ReAct，动作 = 代码块 | 消灭 JSON 协议陷阱；REPL 状态持久 | 仍是 flat loop，不解决长程规划 | **协议层升级（M1）**，与控制结构正交 |
| Plan-and-Execute (arXiv:2305.04091) | Planner 先给 N 步计划 → Executor 顺序执行 | 强制全局思考；plan 可审计；少漂移 | plan 一旦错全错；探索期 plan 空中楼阁 | **强依赖 schema 先验**——M2 全量预注入后 Planner 不再盲规划，契合 |
| Plan-and-Execute + Replan (LLM Compiler arXiv:2312.04511) | 同上 + 失败/新观察后回 Planner 重规划 | 比固定 plan 灵活；保留全局视图 | 实现复杂度↑；replan 频率难调 | **Hard/Extreme 最佳候选**，Medium 题略重但能用 |
| Reflexion (arXiv:2303.11366) | 失败 → 写自然语言反思 → 重试 | 单 session 内自我纠错 | 需要"失败信号"；我们没 oracle，gold 隐藏 | **不适配**——评分黑盒，反思不到根因 |
| ToT / LATS (arXiv:2305.10601 / 2310.04406) | 分支搜索 + 启发回溯 | 谜题/多解空间任务强 | 3–10× token 开销；需 value function | **不适配**——表格题答案唯一，烧 token 不值 |
| 多 Agent (AutoGen / MetaGPT / CAMEL) | Researcher / Coder / Reviewer 多角色 | 分工与互查 | token × N；同模型多角色互补性弱 | 受"单一 LLM"约束**性价比低**；可作 self-review 兜底 |
| Self-Consistency (arXiv:2203.11171) | 同问题 N 次独立采样 → 多数投票 | 单步噪声鲁棒 | 线性 token 成本 | **列 multiset 评分天然支持 cell-level voting**，Easy/Medium N=3 低成本提分 |
| DS-STAR (arXiv:2509.21825) | Schema → Plan → Code → Verify → Refine 五段流水线 | 专为数据科学题设计的 SOTA | 工程量大 | **理论最佳**——本质就是 M1+M2+M3+M4 合体 |

#### 我们的选择：差异化路由的混合架构

不是单一架构通吃，而是按题目难度切换控制粒度（与 [docs/分难度方案设计.md](分难度方案设计.md) "统一骨架，差异在 Executor"思路一致）：

```text
                     ┌─ Easy / Medium ──→ CodeAct flat loop (max_steps=8)
                     │                     + Self-Consistency N=3（cell-level vote）
M2 Schema 预扫 ─→ Difficulty Router ──┤
                     │
                     └─ Hard / Extreme ─→ Plan-Executor with Replan
                                          ├── Planner: 看完整 schema + knowledge.md 出 3–5 步代号化 plan
                                          ├── Executor: 每步 CodeAct REPL 执行（复用 Easy 分支的 executor 代码）
                                          └── Replan trigger:
                                              · 单步连续 3 次失败 → 回 Planner 改路径
                                              · 中间结果不符 plan 预期 → 回 Planner 修剩余步
                                              · 5 次 replan 仍失败 → 强制 submit 当前最佳 df
```

#### 为什么这套搭配在我们约束下最划算

1. **Qwen3.5-35B-A3B 是 MoE（active 3B）**：长链推理弱、并发便宜。Easy 跑 N=3 self-consistency 总成本 ≈ 单跑 35B 一次
2. **256K 原生上下文**：M2 全量 schema 注入后 Planner 不会盲规划，Plan-Execute 最大软肋（plan 空中楼阁）被堵上
3. **CPU-only 评测容器**：所有架构都 inference-only，pandas / duckdb / sqlite 直接跑
4. **列 multiset scoring**：cell-level self-consistency 可直接投票，比 string-level 投票稳得多
5. **统一骨架**：Planner 在 Easy 题退化为"空 plan"（直接进 Executor 单循环），所以**两条分支共享同一份 CodeAct executor 代码**，不是两套实现
6. **死循环止损天然内置**：Replan trigger 包含原 M5（连续 3 次同签名失败）的逻辑，但层级更高——不再是事后补丁，而是架构内建的回退机制

#### 难度路由怎么判定

第一版用 `task.json` 自带的 `difficulty` 字段（`easy / medium / hard / extreme`）硬路由。第二版可以加 LLM-judge 一次性预判（看完 question + schema 估计需要的步数），但只在第一版跑通后再做。

**适用条件**：所有 agent 循环；与 M1 / M2 / M3 全部正交叠加。

**预期**：

- 死循环类 timeout 任务（当前 7 题）从"无提交"变"次优有提交"
- Hard/Extreme 题从 0.51 / 0.00 提升至（暂未量化，需做对照实验）
- 整体 macro Score 提升幅度大于 micro（Extreme 基数小但权重相同）

## 排期

每个 Phase = 内部开发 + 外部验证收尾，不再单独排"DABench 阶段"。

| 阶段 | 内容 | 工作量 | 内层验证（50 demo） | 外层验证（DABench 子集） |
| --- | --- | --- | --- | --- |
| Phase 1 | M1 CodeAct（L1） | ~1 天 | `__error__` 率从 33.7% 降到 <5% | 30 题抽样，相同指标同向 |
| Phase 2 | M2 全量预注入（L2） | ~1 天 | list_dir + read_file 调用数降到 0；平均步数下降 4–6 | 30 题抽样，平均步数同向下降 |
| Phase 3 | M3 归一化 + 形态约束（L3） | ~1.5 天 | "提交了但 Score=0" 题数 <5；Score 提升 | 30 题抽样，Score 提升 |
| Phase 4 | M4 Plan-Executor 难度路由（L4） | ~3 天 | Hard / Extreme Score 提升；timeout 任务清零 | DABench 全量 257 题，看是否在长程任务上同向涨 |

每个 Phase 提交前必须出对照报告：micro Score、macro Score、`__error__` 率、平均完成步数、提交率、按难度 Score 分布——**demo 50 题与 DABench 外部样本同向变化**才算有效。任何只在 demo 涨、DABench 平/跌的，**回滚**——这才是真正的过拟合信号。

## 不做的事

- 单题 prompt 补丁（"如果 question 提到 thrombosis 就……"）
- LLM-as-judge 多模型投票（违反"单一 LLM"硬约束，见 [分难度方案设计.md](分难度方案设计.md) §0）
- 联网检索（评测容器无网络）
- 微调 Qwen3.5（无算力 + 时间不够）
- ToT / LATS 类分支搜索（token 成本不划算，且无好的 value function）
- Reflexion 类自我反思（缺失败信号，反思不到根因）
