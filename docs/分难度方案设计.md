# KDD Cup 2026 DataAgent — 分难度方案设计

> 本文档面向 3 人学生队的实际开发。依据 [比赛简介.md](比赛简介.md)、[方案调研综述.md](方案调研综述.md)、以及 `reports/demo_profile.md` 对 Phase 1 Demo 50 题的实测 profiling。SOTA 引用以 arXiv ID + GitHub 为准，**未核实的标 "待核实"**（CLAUDE.md 要求不编造引用）。
>
> 核心要点：**四个难度共享一个统一骨架（Planner → Executor → Verifier → Formatter），差异只在 Executor 的工具链和 Planner 的 schema-guided 策略**。这样代码复用率最高，团队分工也最清晰。

---

## 0. 设计原则（硬约束回顾）

四档方案必须同时满足：

1. **单一 LLM**：Qwen3.5-35B-A3B（256K 原生上下文，MoE 35B/3B，Apache 2.0）经阿里云百炼 API，**不能换 GPT-4 / Claude 做裁判**；凡涉及 "LLM-as-judge" 的部分一律用 Qwen + self-consistency 替代多模型投票。
2. **CPU-only eval 容器**（16 CPU / 64 GB RAM，无网络）：所有 embedding、BM25、向量库、tokenizer 都必须预先打进 Docker 镜像；pandas/duckdb/sqlite3 为主，禁用任何联网 API（除百炼）。
3. **评分指标**：`Score = Recall − λ·(Extra/Predicted)`，**列名忽略、行序忽略、multiset 比较**。含义：
   - **宁少勿多**：每多一列几乎零收益但扣分，反模式是 "把能想到的都输出"。
   - **形状约束**：任务说 top-5 就必须严格 5 行。
   - **列名 agnostic**：不要浪费 token 纠结列名对齐，**内容语义和类型对齐**才是关键。
4. **聚合方式未公开**（micro 还是 macro 平均未定，见 session 讨论）。对冲策略：本地评测器**同时报告 micro + per-difficulty macro**，两个指标同时优化。

## 1. 统一架构（所有难度共享）

```text
┌─────────────────────────────────────────────────────────────────┐
│                    Unified DataAgent Skeleton                    │
│                                                                  │
│   [1] Task Loader ──→  读 task.json + knowledge.md + context/    │
│                        列出模态（csv/db/json/doc），路由到难度档   │
│                                                                  │
│   [2] Planner       ──→ Schema-guided：产出 {output_schema,       │
│                         sub_tasks, expected_shape, 聚合公式}      │
│                                                                  │
│   [3] Executor      ──→ 按难度档切换工具链（见 §2–§5）             │
│                         共性：Python sandbox 是最终落地器          │
│                                                                  │
│   [4] Verifier      ──→ ① 形状/基数校验（行列数 vs planner 预期） │
│                         ② 值范围/类型合理性（全 NaN / 全 0 告警） │
│                         ③ self-consistency / 反思（触发 repair）  │
│                                                                  │
│   [5] Formatter     ──→ DataFrame → submission.csv                │
│                         强制：去掉中间列、数值 2 位小数、日期 ISO │
└─────────────────────────────────────────────────────────────────┘
```

**核心抽象**：Executor 内部是 Python 沙箱（Program-of-Thoughts 范式，arXiv:2211.12588），SQL / RAG / IE 只是沙箱里的库调用，不是独立执行器。这点见 [方案调研综述](方案调研综述.md) §8.1 的修正说明。

## 2. Easy（30%，15/50）——Python + pandas 表格 QA

**任务特征**（reports/demo_profile.md 实测）：
- 100% 含 `knowledge.md`，80% 含 `csv/`，100% 含 `json/`，**无 DB、无 doc/**
- 输入尺寸中位数 0.3 MB，最大 58 MB → pandas in-memory 完全够用
- gold.csv 行数中位数 1，极少数到 140 行

### 2.1 技术栈

| 组件 | 选型 | 理由 |
|---|---|---|
| Harness | **Qwen-Agent `CodeInterpreter`**（[QwenLM/Qwen-Agent](https://github.com/QwenLM/Qwen-Agent)） | 官方，与 Qwen 对齐；本地 Jupyter 内核、变量持久化、沙箱较成熟 |
| Action 范式 | **CodeAct**（arXiv:2402.01030，[xingyaoww/code-act](https://github.com/xingyaoww/code-act)） | 把 tool call 统一为 Python 代码，对 pandas 天然契合 |
| Planner 增强 | **DS-STAR 的 file analysis module**（arXiv:2509.21825，Google Research, 2025） | 自动对异构 CSV/JSON/Markdown 做类型推断与计划充分性 LLM-judge，比朴素 Planner 显著减少 "漏读文件" |
| 对照 baseline | **DeepAnalyze-8B**（arXiv:2510.16872，人大+清华, 2025，[ruc-datalab/DeepAnalyze](https://github.com/ruc-datalab/DeepAnalyze)） | 端到端数据 agent 开源实现，8B 模型即超多数闭源；我们不能换模型，但其 **agentic 轨迹合成方法论** 可用作自造 few-shot 库 |
| 自修复 | **Self-Debug**（arXiv:2304.05128）+ **Reflexion**（arXiv:2303.11366） | 最多 3 轮（超过 3 轮基本不再收敛，徒增 token） |
| 沙箱 | `subprocess` + `resource.setrlimit` + 超时 30s | 防死循环 / OOM |

**不选 OpenHands / OpenDevin**：Docker-in-Docker 假设对 eval 容器不友好，且对纯表格 QA 过重。
**不选 PandasAI**：prompt 硬编码、不方便自定义 verifier，只用于 baseline 对照。

### 2.2 Pipeline

```text
Planner（Qwen）
  输入：task.md + knowledge.md + 对每个 csv/json 执行 df.head(5) + df.dtypes + df.describe()
  输出：JSON {
          output_schema: [{name, dtype, semantic}, ...],  # 仅用于 verifier，不输出到 CSV
          expected_rows: int | "variable",
          plan: ["step 1: ...", "step 2: ...", ...]
        }

Executor（CodeAct loop，最多 5 轮）
  for step in plan:
      code = Qwen(df.head, history, step)
      result = sandbox.exec(code)
      if error: self_debug(code, traceback) → retry ≤3
  final = 最后一步的 DataFrame

Verifier
  检查 final.shape vs expected_rows；异常值扫描（全 NaN、负数但 schema 说 positive）
  若失败 → Reflexion：重新规划

Formatter
  去掉中间计算列；数值 round(2)；日期 strftime("%Y-%m-%d") → submission.csv
```

### 2.3 常见陷阱 + 缓解

| 陷阱 | 缓解 |
|---|---|
| 输出多余列（groupby 后 reset_index 把 group key 也带进来） | Planner 显式声明 output_schema；Formatter 按 schema 投影 |
| 日期时区 | `pd.to_datetime(..., utc=True).dt.tz_convert(None)` 统一去时区 |
| NaN 语义（sum 对全 NaN 返回 0） | Verifier 对 "全零列" 发告警，让 Planner 确认 |
| 浮点精度 | Formatter `.round(2)`，与 gold 保持同粒度 |

### 2.4 本地评测策略

- **Sanity 集**：挑 demo 里 5 题 easy 跑到 100% 通过
- **Dev 集**：剩下 10 题 demo easy + 5 题自造同构题
- **优化指标**：`Score` + 通过率；重点关注 "多预测列惩罚" 是否被压住

## 3. Medium（46%，23/50）——Text-to-SQL on SQLite

**任务特征**：
- 100% 含 SQLite DB；74% 含 CSV、57% 含 JSON（多源混合）
- 输入尺寸 median 1.4 MB，但 extreme 值到 440 MB（重型 DB）
- 这是占比最大的档，**Medium 的提升 = 整体排名的基本盘**

### 3.1 技术栈

| 组件 | 选型 | arXiv |
|---|---|---|
| **首选 — 纯推理** | **RSL-SQL**（2025）：双向 schema linking + 二选一策略 + 多轮 self-correction；BIRD 67.2% (GPT-4o) 纯 prompt 级无训练 | 2411.00073，[Laqcce-cao/RSL-SQL](https://github.com/Laqcce-cao/RSL-SQL) |
| 骨架参考 | **CHESS** 的四模块（Information Retriever / Schema Selector / Candidate Generator / Unit Tester） | 2405.16755 |
| 候选生成 | **CHASE-SQL** 的多路径 CoT（divide-and-conquer + query plan + online synthetic） | 2410.01943 |
| 候选生成（规模化，可选） | **Agentar-Scale-SQL**（蚂蚁, 2025）：Orchestrated Test-Time Scaling + 并行锦标赛选择；BIRD test **81.67% 当前 #1** | 2509.24403，[antgroup/Agentar-Scale-SQL](https://github.com/antgroup/Agentar-Scale-SQL) |
| 选号 | **Self-consistency execution voting**（N=6，温度 0.5）+ pairwise Qwen judge 兜底 | 2203.11171 (SC) |
| Schema linking | RSL-SQL 双向 linking 优先；fallback 到 CHESS LSH + value preview、MAC-SQL Selector | 2312.11242 |
| 诊断基准 | **SWE-SQL / BIRD-CRITIC-1**（NeurIPS 2025）：600 dev + 200 OOD SQL 诊断题 | [bird-bench/BIRD-CRITIC-1](https://github.com/bird-bench/BIRD-CRITIC-1) |
| 兜底 | **DAIL-SQL** skeleton-based example selection | 2308.15363 |

**不微调 Qwen**（API-only，没法改底层），所以 **DTS-SQL / CodeS** 的模型本身不能用，但其 "先 schema 再 SQL" 拆分思想可借鉴。

### 3.2 Pipeline

```text
[Offline / 任务加载时]
  DB Profiler：
    - sqlite_master 拉全 schema
    - 每张表 SELECT * LIMIT 5 取样本
    - 对字符串列建 BM25 倒排索引（rank_bm25，纯 Python）
    - 对枚举型列记录 distinct values
    → db_profile.json

[Online / 每题]
  [1] Information Retriever（Qwen）
      从 question 抽取字面量 → BM25 命中 → 得到 value_hits
  [2] Schema Selector（Qwen）
      输入：question + full schema + value_hits
      输出：相关表/列子集（JSON）
  [3] Candidate Generator（Qwen × N=6，温度 0.5）
      三种 CoT 风格混采：
        - divide-and-conquer（子问题 → 子 SQL → 合成）
        - query plan CoT（先写执行计划再写 SQL）
        - synthetic example prompt（Qwen 先造 2 个 schema 内的例子再写）
  [4] Executor
      每条候选 run → {rows, cols, error}
      剔除报错；对成功结果做 multiset hashing
  [5] Selector
      ① 多数投票（频次最高的 result hash）
      ② 若前 2 名并列 → pairwise Qwen judge（给两条 SQL + 两个结果，选更合理的）
  [6] Repair（若最终结果为空或形状不合理）
      把 SQL + 空结果给 Qwen，让它放宽 WHERE / 换 JOIN 方向 → 重生成
```

### 3.3 常见陷阱 + 缓解

| 陷阱 | 缓解 |
|---|---|
| JOIN 方向错 | DB Profiler 把 FK 显式列进 schema 提示 |
| 聚合维度少/多一列 | Planner 预声明 output_schema，SQL 生成时对齐 |
| 日期字符串（SQLite 无原生 DATE） | Schema 里标注 `TEXT (date-like)`；prompt 例里强制 `strftime` |
| 大小写 / 空格 | 所有字符串比较默认 `COLLATE NOCASE` |
| Self-consistency 被系统性偏差淹没 | CHASE-SQL 的 pairwise judge 作二选一兜底 |
| BIRD 风格 `evidence` 列未利用 | 若 `knowledge.md` 里含业务规则（如 "计算毛利率 = ..."），先提炼成 SQL CASE 注释塞进 Candidate Generator prompt |

### 3.4 本地评测策略

- **Sanity 集**：5 题（覆盖单表 / 双表 join / 聚合 / 子查询 / 日期）
- **Dev 集**：剩下 18 题 demo medium + 用 `scripts/convert_bird.py` 转 BIRD 30 题（balanced by difficulty）
- **关键监控**：SQL 执行成功率、self-consistency agreement 率（低 agreement 就是潜在翻车）

## 4. Hard（20%，10/50）——doc/ 叙事陷阱 + IE → 聚合

**任务特征**（reports/demo_profile.md）：
- 100% 含 doc/（12K–174K 字节 Markdown 长文档）
- 100% 含 knowledge.md，70% 含 csv/，30% 含 db/
- 文档里埋 "修正值" 陷阱：`task_418` 的 "initially logged as 35.0 U/L; however, a calibration review led to a corrected value of 28.0 U/L"

### 4.1 技术栈

| 组件 | 选型 | 理由 |
|---|---|---|
| **首选 — 竞赛冠军方案** | **HEAR: Holistic Extraction and Agentic Reasoning**（ACM MM 2025 LAVA Grand Challenge **1st Prize**，SYSUpporter/中山大学） | 闭环耦合"整体文档解析 + 多 agent 推理"——与 KDD Cup Hard 层"长 Markdown → 聚合"的子问题高度对齐。ACM DOI 10.1145/3746027.3761999 |
| 主框架 | **DocETL**（arXiv:2410.12189，[ucbepic/docetl](https://github.com/ucbepic/docetl)） | 声明式 map/reduce/resolve/split/gather，`resolve` 算子天然处理 "同一实体多处提及" 冲突 |
| DocETL 2025 升级 | **MOAR**（arXiv:2512.02289，Berkeley 同 DocETL 团队, 2025） | 新优化器含 >30 条 directives，对比 ABACUS 准确率 **+27%**、成本 **−55%**（待仓库发布后直接替换 DocETL 优化器） |
| 文档结构化 refiner | **LongRefiner**（arXiv:2505.10413，ACL 2025） | 利用文档固有层级结构 + 即插即用 refiner，计算成本/延迟 **降低 10×**，对 CPU-only eval 极友好 |
| Routing 思路 | **db3 队 CRAG 2024 冠军方案**（arXiv:2410.00005，[USTCAGI/CRAG-in-KDD-Cup2024](https://github.com/USTCAGI/CRAG-in-KDD-Cup2024)） | "问题先分类再走差异化 pipeline" 是近两届 KDD Cup 冠军共识；每种 Hard 子类型（病历 / 财报 / 代码社区）单独一条链 |
| 表格推理 | **Chain-of-Table**（arXiv:2401.04398）思想（无需完整 GitHub 复现，prompt 化即可） | 把 IE 后的中间表当作可被 "操作链" 推理的对象 |
| 长上下文读图 | Qwen3.5-35B-A3B 直接读（256K 够装 174K 字节） | 无需 RAG，全文一次喂入 |
| 抗 "lost-in-the-middle" | 分段复核（arXiv:2307.03172） | 抽完后对关键段落二次 check |
| Refusal 训练的 prompt 化 | **db3 CRAG-MM 2025 Grand Prize**（arXiv:2509.09681） | 原方案用 SFT+DPO+RL 做 "无证据则拒答"——我们不能微调，但可在 Extractor prompt 硬编码 "若 source_span 无法锚定则输出 null"，直接砍掉多重集指标里的 Extra 惩罚 |

**DocETL 适配**：默认用 OpenAI client，**需要替换成百炼 SDK**。这是 S2 第一周的工作项之一。

### 4.2 Pipeline

```text
[1] Schema Inducer（Qwen）
    输入：task.md + knowledge.md
    输出：{
      target_schema: [{name, dtype, unit, description}, ...],
      aggregation_rule: "按 patient_id 聚合，取最后一次检测值",
      correction_keywords: ["corrected to", "revised", "final value", "修正后", ...]
    }

[2] Doc Chunker
    按 Markdown 结构（##/###）切，保留 section_path
    不按字符硬切（会切断实体）

[3] Extractor（DocETL map 算子，per chunk）
    prompt 要求：
      - 按 target_schema 输出 JSON 列表
      - 每条带 source_span（原文字节区间）和 confidence（high/medium/low）
      - 若同一字段在本 chunk 出现多次，全部列出

[4] Resolver（DocETL resolve 算子，跨 chunk 去重 + 冲突消解）
    规则：
      ① 若出现 correction_keywords → 取 corrected 后的值
      ② 否则按时间戳取最新
      ③ 否则按 confidence 降序取 high
    兜底：Qwen pairwise judge（给两个候选 + 它们的 source_span，问哪个是最终值）

[5] Aggregator（pandas）
    按 aggregation_rule 跑 groupby / merge

[6] Verifier
    ① 形状 check
    ② 回查：随机抽 3 行，让 Qwen 核对 source_span 能否在原文找到 → 不行 → 降置信度 → 重抽
    ③ coverage check：让 Qwen 估算是否有遗漏实体（"是否所有 patient 都被抽到了？"）
```

### 4.3 "修正值" 陷阱专项设计

| 层次 | 做法 |
|---|---|
| 抽取阶段 | 保留 **所有** 候选值（不做早决策），每条带 span + 上下文句子 |
| Resolver prompt | 硬编码触发词（见 §4.2 Schema Inducer 输出）；例 demo：先给 3 个修正值例子 |
| 时序 fallback | 若 doc 有时间戳（`2024-03-01`），按时间降序 |
| 置信度 | 低置信项（e.g. 单一未重复提及）进二轮 Qwen 复核 |
| 可验证性 | Verifier 强制 source_span 必须回查在原文，打击幻觉 |

### 4.4 常见陷阱 + 缓解

| 陷阱 | 缓解 |
|---|---|
| 幻觉数值 | 强制 source_span + 后处理校验 span 确实在原文 |
| 单位不一致（mg/dL vs mmol/L） | Schema 带 unit，聚合前归一（查 `knowledge.md` 的换算） |
| 别名（"Pt. A" vs "Patient A"） | DocETL resolve + Qwen 相似度打分 |
| 漏抽中段 | 抽完后按 section_path 统计覆盖率；空段落触发定向再抽 |
| 过度引用（照搬原文句子当字段值） | prompt 例里强调 "只输出结构化值，不要原文句子" |

### 4.5 本地评测策略

- **Sanity 集**：3 题（必含 task_418 这种修正值陷阱）
- **Dev 集**：剩下 7 题 demo hard + 合成 10 题（见 §7.3 自对弈）
- **关键指标**：抽取 F1（per-entity）+ 最终 Score；监控 correction trap 的识别率

## 5. Extreme（4%，2/50）——超长文档 (up to 1M bytes)

**任务特征**：
- 仅 2 题样本（task_418 extreme: 363K doc；task_420 extreme: 1M doc）
- 1M 字节 UTF-8 中文约 330K tokens，**超过 Qwen 原生 256K 上限**
- 与 Hard 同模态，但量级上 RAG 不可避免

### 5.1 技术栈

| 组件 | 选型 | arXiv |
|---|---|---|
| **首选 — 竞赛冠军方案** | **AdaDocVQA**（ACM MM 2025 LAVA **2nd Prize**，arXiv:2508.13606，[Haoxuanli-Thu/AdaDocVQA](https://github.com/Haoxuanli-Thu/AdaDocVQA)） | 三件套高度契合 Extreme 的 CPU-only 约束：① 混合文本检索的文档切分；② 自动生成高质量推理 QA 对的数据增强；③ **自适应 ensemble + 动态配置 + early-stopping**（低资源下 LAVA 上 59% acc，目前最强公开方案） |
| 文档结构化 refiner | **LongRefiner**（arXiv:2505.10413，ACL 2025） | 利用 Markdown 天然层级做结构感知切块 + 即插即用 refiner，**延迟降 10×** |
| 检索 | Hybrid：**BM25 (`rank_bm25`)** + **本地 dense (bge-m3 / bge-small-zh)** | — |
| Retrieval 策略 | **LongRAG** 思想（大 chunk=4K tokens） | 2406.15319 |
| 分层摘要 | **Context-Aware Hierarchical Merging**（arXiv:2502.00977, 2025）+ 两层 map-reduce | RAPTOR 思想的 2025 验证版（Llama 3.1 8B/70B 实测） |
| 多 agent 协作 | **LongAgent**（arXiv:2402.11550）基线；**DS-STAR**（arXiv:2509.21825, 2025）file analysis module | 超 128K 时先提取结构 context，再分发给 extractor |
| 子查询分解 | **IRCoT**（arXiv:2212.10509）+ **Self-Ask**（arXiv:2210.03350） | — |
| 对比基线 | MemGPT / Letta 思想的简化版 | 2310.08560 |

**不用 LLMLingua**：需要额外压缩模型，本地 RAM / 成本权衡不划算。
**不完整复现 RAPTOR**：预处理阶段用 Qwen 递归摘要成本高，2 层 map-reduce 已覆盖 80% 收益。

### 5.2 Pipeline

```text
[任务加载时 / 一次性]
  Indexer：
    - 按 Markdown 结构 + 字符上限（4000 chars）切 chunk
    - BM25 索引 + bge-m3 embedding（模型预打进 Docker，~2.3 GB）
    - 可选：分层摘要（对每个 chunk 摘 1 次，再对摘要合并摘 1 次）
      顶层摘要 <10K tokens 可直接塞 Qwen 上下文

[Online]
  [1] Query Decomposer（Qwen）
      question → sub_questions[]（最多 2 层递归）

  [2] For each sub_question:
      a) 先查顶层摘要 → Qwen 判断该 sub_q 需要哪些章节
      b) Hybrid retrieval（BM25 + dense） top-K=20 → Qwen pairwise rerank → top-5
      c) 把 top-5 chunk 拼进 256K 窗口（预计 <50K tokens）
      d) Schema-guided Extractor（同 Hard §4.2 [3]）

  [3] Cross-subquestion Aggregator
      合并所有 sub_q 的抽取结果 → pandas

  [4] Coverage Check
      让 Qwen 检查：是否所有 sub_q 都已回答？是否有遗漏实体？
      若否 → 扩检索 K → 20 → 50 再跑一轮

  [5] Verifier + Formatter（同 Hard）
```

### 5.3 摘要 prompt 要点

**关键**：摘要时必须保留数值、日期、实体，否则 extreme 级别的细节题直接失分。

```text
你在压缩一段医学病历文档。保留全部：
  - 数值（检测值、剂量、天数）
  - 日期、时间戳
  - 人名、药物名、疾病名
  - "修正"、"撤回"、"最终"、"corrected"、"revised"、"final" 等线索词
删除：
  - 寒暄、免责声明、格式化标题
  - 重复陈述
控制在原长度的 30%。
```

### 5.4 陷阱 + 缓解

| 陷阱 | 缓解 |
|---|---|
| 关键细节被摘要吞掉 | 摘要 prompt 强调保留数值/日期/实体 |
| 检索漏召 | BM25 抓字面 + dense 抓同义，保留并集 |
| 子问题爆炸 | 硬限递归深度 2 层 + token budget |
| 跨段实体链接错 | 全局 resolver 阶段（同 Hard Resolver） |
| bge-m3 CPU encode 太慢 | 预处理阶段 batch encode；eval 时只对 query encode（单次 <1s） |

### 5.5 本地评测策略

- **Sanity 集**：仅有 2 题 demo extreme 可用，必须两题都通
- **Dev 集扩充**：合成 5–10 题超长文档题（见 §7.3）
- **成本监控**：extreme 单题 token 成本预计 ¥0.5–¥2，Phase 1 全期总成本需控住

## 6. 评估方法

### 6.1 本地评分器（S2 第一周交付）

核心函数签名：

```python
def score_task(pred_csv: Path, gold_csv: Path, lam: float = 0.5) -> dict:
    """返回 {'recall': float, 'extra_ratio': float, 'score': float, 'match_detail': list}"""
    ...

def score_batch(pred_dir: Path, gold_dir: Path) -> dict:
    """返回 {
        'micro_mean_score': float,             # 所有任务直接平均
        'macro_mean_score': float,             # 按难度分桶求均值再等权相加
        'per_difficulty': {easy/medium/hard/extreme: {mean, min, max, count}},
        'fail_list': [task_id, ...],
    }"""
```

**实现要点**：
1. 列 multiset 比对：对每列的值向量，**转为归一化字符串后做 Counter 等值比较**
2. 值归一化：数字 round(2)；日期 ISO；NaN/None 统一为 `""`
3. 浮点 tolerance：`abs(a-b) < 1e-6` 允许
4. 列名 **不参与比对**（只用于调试日志）
5. 与 Starter Kit 官方评测脚本在所有 demo 任务上对齐（偏差 0 分）

### 6.2 三层测试切分

| 层 | 规模 | 用途 | 来源 |
|---|---|---|---|
| **Sanity** | 10–15 题（4 难度均覆盖） | 每次 commit 前跑；必须 100% 通过 | demo 精选 |
| **Dev** | 50–80 题 | 迭代优化的主战场；每日跑 | demo 剩余 + BIRD 转换 + 合成题 |
| **Held-out** | ≥100 题 | 冻结直到提交前，防 dev 过拟合 | 合成题为主，人工抽查 |

详见 [方案调研综述.md](方案调研综述.md) §附录 B 与 `scripts/convert_bird.py`。

### 6.3 指标对冲（对聚合方式未知的风险）

每次迭代输出 4 个数：

```text
Overall:
  micro mean   : 0.623  # 所有任务直接平均（猜测的默认聚合）
  macro mean   : 0.581  # 按难度等权（另一种常见聚合）
Per-difficulty:
  easy    : 0.80 (n=15)
  medium  : 0.65 (n=23)
  hard    : 0.35 (n=10)
  extreme : 0.25 (n=2)
```

**决策规则**：
- 两个 overall 都单调上升 → 继续
- 只有 micro 上升但 macro 下降 → 说明在强化已通过的题而牺牲 hard/extreme，停手重看
- 反之亦然 → 考虑是否值得（hard/extreme 只 12 题，方差大）

## 7. AI 驱动的迭代优化

手调 prompt 边际收益递减，越往后越要自动化。

### 7.1 主框架：DSPy + **GEPA**（2025 新 SOTA，严格优于 MIPROv2）

- **DSPy**（arXiv:2310.03714，[stanfordnlp/dspy](https://github.com/stanfordnlp/dspy)）：把每个难度的 pipeline 写成声明式 `Module`，每个阶段（Planner / Schema Selector / Extractor / Verifier）声明 `Signature`
- **GEPA: Reflective Prompt Evolution**（arXiv:2507.19457，**ICLR 2026 Oral**，[gepa-ai/gepa](https://github.com/gepa-ai/gepa)）：反射式进化优化器，用文字 feedback 引导 prompt 演化
  - **与 MIPROv2 对比**：AIME-2025 上 **+12%**，综合 **+13%**（MIPROv2 仅 +5.6%）
  - **已集成到 DSPy**（`dspy.GEPA`），API 与 MIPROv2 近似，迁移成本低
  - 只需少量 labeled 数据，**适合竞赛 few-shot 设定**
- **MIPROv2**（arXiv:2406.11695）仅作 fallback 基线
- **评测函数直接用 `score_task`**（§6.1），优化方向与最终评分一致

**落地步骤**：
1. 把 baseline（手调版）的每个 stage 改写为 DSPy Module
2. 用 dev 集 50 题作为优化集
3. 先跑 `BootstrapFewShot`（便宜）作下限基线
4. 主优化跑 **GEPA**（而非 MIPROv2），预算控在 ¥200 以内
5. 每难度单独优化 + 联合优化都试，选 held-out 上更好的

### 7.2 轻量 fallback：TextGrad + OPRO

- **TextGrad**（arXiv:2406.07496）：pipeline 结构稳定时只调 prompt 文本
- **OPRO**（arXiv:2309.03409）：最简的 "让 LLM 自己改 prompt" 范式，几十行可实现

DSPy 太重时用。

### 7.3 错误聚类 → 数据扩充循环

这是 2024–2025 工程上最见效的范式：

```text
┌─────────────────────────────────────────────────────────────┐
│  Loop: run → cluster → augment → retrain (prompt)            │
│                                                              │
│  [1] 跑 dev 集，收集 {task_id, trace, pred, gold, score}     │
│  [2] Qwen-as-judge 给每条失败打标签                          │
│      （"SQL JOIN 方向错" / "抽取漏单位" / "修正值未识别"...） │
│  [3] 按标签聚类（直接 Qwen 归并，或 embedding + KMeans）     │
│  [4] 对薄弱类别（占失败 >20%）：                             │
│      - Qwen 基于已有失败题合成 10–20 条变体（带 gold）       │
│      - 加入 dev 集                                           │
│  [5] DSPy / 手工迭代 → 回归 held-out                         │
│  [6] GOTO [1]（每周 1 次循环）                               │
└─────────────────────────────────────────────────────────────┘
```

参考：
- **AutoBencher**（arXiv:2407.08351 待核实）：自动发现 benchmark 盲点
- **EvalGen**（arXiv:2404.12272 待核实）：自动合成评测准则
- ~~ADAS / AFlow~~：自动搜 agent 架构，对 3 人队偏重，Phase 2 冲刺再考虑

### 7.4 Reflexion 轻量层（每题内）

不等 dev loop 结束，**每题内**加 1 轮反思：
- Executor 跑完 → Verifier 报警 → Qwen 对失败原因写一段 reflection → 重跑一次
- 最多 1 轮，避免 token 爆炸

详见 Reflexion (arXiv:2303.11366)、Self-Refine (arXiv:2303.17651)。

### 7.5 实操注意

| 坑 | 缓解 |
|---|---|
| 过拟合 dev | 保留 held-out，限制 MIPRO 迭代次数；每次优化后必跑 held-out |
| 优化成本爆炸 | 先小规模预实验定超参；每周 API 成本不超过 ¥100 |
| Judge 偏差（Qwen 评自己） | Judge prompt 强制引用客观证据（执行结果、原文 span）；定期人工抽查 judge 一致性 |
| 合成题质量 | 合成后人工抽 10% 复核，不合格丢弃 |

## 11. 2025-2026 升级清单（本次新增）

> 用户提问 "有 25-26 年的最新方案吗，特别是一些相关比赛得奖的项目，比如 hard 类的题目和 ACM MM LAVA 赛题" 触发本次升级。核实结果：**LAVA 是 ACM MM 2025**（非 2024），冠亚军方案高度可迁移。

### 11.1 LAVA Grand Challenge 2025 启示

LAVA（Duc Minh Vo et al., SoftBank + 东京大学 组织）本质是日英双语多页 PDF 上的 10 路多选 VQA，**强调跨页面/跨文本-图表的证据整合**。与 KDD Cup Hard 层 "长 Markdown → 聚合" 的共性：
- 都是长文档 + 多段证据融合
- 都是低资源约束（LAVA 要求无外部检索，KDD 要求 CPU-only + 单 LLM）
- 都是结构化输出（LAVA 10 选 1，KDD 列多重集）

#### 1st Prize — HEAR（§4.1 已加）

**H**olistic **E**xtraction + **A**gentic **R**easoning，中山大学 SYSUpporter 队。核心思想：**整体解析 → 多 agent 推理**，而不是 "RAG + 一次性 answer"。对我们的意义：
- Hard 层的 **Extractor（§4.2 [3]）应该先做整体 schema 解析再并行多 agent 抽取**，而不是 chunk-by-chunk 顺序抽取
- 多 agent 之间用 **冲突检测 + pairwise judge** 而不是简单投票

#### 2nd Prize — AdaDocVQA（§5.1 已加）

Haoxuan Li et al.（arXiv:2508.13606，[GitHub](https://github.com/Haoxuanli-Thu/AdaDocVQA)），**CPU-only 友好**，LAVA 数据集 59% acc。三件套：
1. **混合文本检索的文档切分**——直接用于 Extreme 层 chunking
2. **自动生成高质量推理 QA 对**——正好用于 §7.3 的合成数据循环
3. **自适应 ensemble + 动态配置 + early-stopping**——解决 Agentar-Scale-SQL "并行锦标赛太贵" 的问题

**强烈建议 S2 在 Week 1 D5-D7 期间精读其代码**。

### 11.3 历届 KDD Cup 冠军的共同模式

从 **db3 队**（CRAG 2024 Grand Prize + CRAG-MM 2025 Grand Prize）、**NVIDIA**（Amazon KDD Cup 2024 全赛道冠军）的获奖方案中提炼的三条工程共识：

1. **Routing-based pipeline**：问题先分类，不同子类型走不同 pipeline（db3 CRAG 2024 的 Task 1/2/3 用三套策略）。我们的方案已按难度路由，**进一步可在 Hard 层按 doc 子类型再分叉**（病历 / 财报 / 代码社区）。
2. **数据合成弥补样本稀缺**：NVIDIA 用 Llama 3 70B 合成题训练 LoRA；我们不能微调，但可把合成题用于 **few-shot pool + dev 集扩充**（§7.3）。
3. **Refusal 控制幻觉**：db3 CRAG-MM 2025 用 SFT+DPO+RL 训练 "无证据则拒答"。我们在 Extractor prompt 硬编码同样规则（§4.1 末行），**直接压 Extra 惩罚**——对 multiset 指标是系统性优化。
