## 0. 赛题方案约束回顾（用于筛选技术选型）

1. **评测阶段唯一可用的推理 LLM**：官方部署的 `Qwen3.5-35B-A3B`，通过 HTTP 调用，不能换模型。
   - HF 官方卡 (<https://huggingface.co/Qwen/Qwen3.5-35B-A3B>)：**总 35B / 激活 3B**；MoE，256 专家，每次激活 8 routed + 1 shared；40 层，hidden 2048。
   - **原生多模态**（Image-Text-to-Text + 视频理解，MMMU 81.4 / VideoMME 86.6）——Phase 2 的图像/视频可以直接喂入，无需外挂 VLM。
   - 原生上下文 **262,144 tokens**，YaRN 可扩到 ~1M；官方推荐至少 128K。
   - 默认启用 thinking 模式（输出含 `<think>...</think>`），推荐输出长度 32K，复杂任务可至 80K。
2. **Agent 运行环境：无 GPU、16 核 CPU、64 GB RAM、12 h 总预算**：embedding / 重排器必须 CPU 友好；大模型的调用次数是**核心预算**。
3. **输出：列签名匹配评分**（`Score = Recall − λ · Extra/Predicted`）：任务本质是 **"给定复杂问题，返回一张结构化表"**。
4. **输入：异构多模态**：CSV、关系型 DB、JSON、短～128K 长文档，Phase 2 加入 data image / 视频。
5. **交付形式**：离线 Docker 镜像，评测节点无外网。

任何不满足上述 5 点的技术，**都不能出现在最终提交方案**里。

## 1. 问题抽象：Data Agent 的 4 个子能力

把赛题拆成 Agent 的能力原子，便于映射到文献：

| 子能力 | 一句话定义 | 在本赛题的映射 |
|--------|-----------|-----------------|
| **Task Decomposition & Planning** | 把用户自然语言问题拆成可执行子步骤 | 解析任务 prompt → 决定要查哪张表、哪段文档 |
| **Tool Use / Grounded Execution** | 调用 SQL、Python、检索 API 等外部工具完成子步骤 | 生成 SQL 查 DB、跑 Pandas 统计、向量检索长文档 |
| **Heterogeneous Data Reasoning** | 在结构化 + 非结构化 + 多模态上做联合推理 | 跨表 JOIN + 文档中的数字 + 图表里的数值 |
| **Answer Synthesis & Verification** | 把中间结果汇总成结构化答案，并做一致性检查 | 合成 prediction.csv，校验列、类型、缺失值 |

接下来 §2–§5 按这 4 个子能力分别综述，§6 回到端到端的 Data Agent 系统，§7 是评测与基准调研。

## 2. Agent 范式：Planning 与 Tool Use

### 2.1 单 Agent 的基础范式

- **ReAct** (Yao et al., 2023, arXiv:2210.03629)：Thought → Action → Observation 循环，被几乎所有后续 Agent 使用，也是 **本赛 Starter Kit 的基线**。
- **Reflexion** (Shinn et al., 2023, arXiv:2303.11366)：引入自我反思记忆，能在固定预算下显著提升成功率。对"Phase 1 每天 1 次提交"这种少样本反馈场景特别契合。
- **Self-Refine** (Madaan et al., 2023, arXiv:2303.17651)：同一个 LLM 既生成又批改，无需外部 critic。在 CPU 预算下是最容易落地的自我校验方式。
- **Tree of Thoughts** (Yao et al., 2023, arXiv:2305.10601)：把推理展开成搜索树。**代价高，不建议在 12 h 预算内默认开启**，可在高难度 task 上按 budget 分层使用。
- **Plan-and-Solve / Plan-and-Execute** (Wang et al., 2023, arXiv:2305.04091)：先全局规划一次，再顺序执行。LangChain/LangGraph 的 "PlanAndExecute" 家族采用此范式。相比纯 ReAct，**减少 LLM 调用次数**，更省预算。
- **DSPy** (Khattab et al., 2023, arXiv:2310.03714)：把 prompt 视作可编译的 program，支持自动 prompt 优化。**可在本地开发期用来搜索 prompt**，注意最终镜像无法联网调用外部优化器，需固化成 prompt 模板。

### 2.2 工具学习（Tool Learning / Function Calling）

- **Toolformer** (Schick et al., 2023, arXiv:2302.04761)：让 LLM 自监督学会调用工具的经典工作。
- **ToolLLM / ToolBench** (Qin et al., 2023, arXiv:2307.16789)：大规模 API 调用训练集，论证开源 LLM 的工具调用能力。
- **Gorilla** (Patil et al., 2023, arXiv:2305.15334)：API 调用的指令微调，给出了"工具召回 + 正确调用"的分解评测视角。
- **Qwen-Agent / Qwen2.5 Function Calling**（官方文档，Qwen 技术报告 arXiv:2412.15115 *待核实*）：Qwen 系列原生支持 function calling 与 tool 子语言。**Qwen3.5-A3B 如果完整继承该能力，是 Agent 调度的天然接口**，建议学生 S1 在基线阶段直接用 function calling 而非自造 ReAct 文本协议。

### 2.3 多 Agent 与协作

- **AutoGen** (Wu et al., 2023, arXiv:2308.08155)：多 Agent 对话 + Conversable Agent API，社区最活跃。
- **MetaGPT** (Hong et al., 2023, arXiv:2308.00352)：SOP 思想，把软件开发流程注入 Agent 角色。
- **TaskWeaver** (Qiao et al., 2023, arXiv:2311.17541)：微软提出的 code-first Agent 框架，天然支持数据分析场景。

> **技术选型建议**：默认采用 **Plan-and-Execute + 最多 2 层 ReAct**，不在 Phase 1 引入多 Agent；Phase 2 再按需要增加"规划 Agent / 执行 Agent / 校验 Agent"的分工。多 Agent 最大的成本是 LLM 调用轮次，直接打穿 12 h 预算。

## 3. 结构化数据上的分析：NL2SQL 与 Table Understanding

Data Agent 最重的子任务是"生成 SQL 或 Pandas 查询"。这一块的研究比 Agent 更成熟。

### 3.1 基准数据集

- **Spider** (Yu et al., 2018, arXiv:1809.08887)：跨域单轮 NL2SQL 经典 benchmark。
- **BIRD** (Li et al., 2023, arXiv:2305.03111)：大规模、脏数据、多表、复杂 SQL，更接近真实数据分析场景。**本赛的 structured 部分风格与 BIRD 最接近**。
- **Spider 2.0** (2024, arXiv:2411.07763 *待核实*)：引入企业级真实 SQL 复杂度，多方言、多库。
- **ArcadeNL2SQL / SParC / CoSQL**：多轮对话式 NL2SQL，不是本赛主线。

### 3.2 LLM-based NL2SQL 方法

- **DIN-SQL** (Pourreza & Rafiei, 2023, arXiv:2304.11015)：把问题分类（Easy/Medium/Hard）后再生成，当时 Spider 最强零样本方法之一。
- **C3-SQL** (Dong et al., 2023, arXiv:2307.07306)：三段式 prompt：Clear prompting、Calibration、Consistent output。
- **DAIL-SQL** (Gao et al., 2023, arXiv:2308.15363)：系统性消融 in-context example 的选择与组织策略，当时 Spider/BIRD SOTA 级。
- **MAC-SQL** (Wang et al., 2023, arXiv:2312.11242 *待核实*)：多 Agent NL2SQL，显式分配 "Selector / Decomposer / Refiner"。
- **CHESS / E-SQL** 等 2024 年的若干变体——共同趋势是 **schema linking 离线化 + 预生成 SQL plan + 执行反馈回流**。

### 3.3 Table Understanding（表格作为 context）

- **TaBERT / TAPAS** (Herzig et al., 2020, arXiv:2004.02349 *待核实*)：早期结构化 encoder，在 CPU 上做表格嵌入仍然实用。
- **TableGPT2** (Su et al., 2024, arXiv:2411.02059 *待核实*)：面向表格任务的 LLM，提供了对比基线。
- **Chain-of-Table** (Wang et al., 2024, arXiv:2401.04398)：让 LLM 在表格上做"逐步变换链"，在 TabFact/WikiTQ 等上涨点显著。**思想直接可迁移到本赛：先对大表做投影/过滤链，再做最终聚合**。
- **Table-GPT** (Li et al., 2023, arXiv:2310.09263 *待核实*)：微软提出的表格任务通用指令微调。

### 3.4 大表 / 大 schema 的处理

- **schema linking**：当 DB schema 超过 LLM 上下文时，必须先做相关表/列召回。常见做法：
  1. 列名 / 列描述的 embedding 检索
  2. 基于问题关键字的 BM25
  3. 两者级联 + LLM 精排
- **value retrieval**：把每一列的代表值（如 top-k 值、值域范围）压入 prompt，显著减少 SQL 生成的值错误。DIN-SQL、DAIL-SQL 均验证过价值。

> **本赛建议**：S2 负责实现一套"schema linking + value retrieval"预处理器；SQL 生成采用 **DAIL-SQL 风格 prompt + Chain-of-Table 的分步过滤** 组合；执行用 SQLite/DuckDB 本地跑，结果回灌到 Agent 做自检。

## 4. 非结构化 / 长文档上的 RAG

### 4.1 基础

- **RAG** (Lewis et al., 2020, arXiv:2005.11401)：检索+生成的奠基工作。
- **FiD** (Izacard & Grave, 2021, arXiv:2007.01282 *待核实*)：Fusion-in-Decoder，多文档融合的经典。
- **ColBERT / ColBERTv2** (Khattab & Zaharia, 2020–2021, arXiv:2004.12832, arXiv:2112.01488)：late-interaction 检索，**在无 GPU 环境下也可用其压缩版本，召回质量远高于 BM25**。

### 4.2 近 2 年的进展

- **Self-RAG** (Asai et al., 2023, arXiv:2310.11511)：LLM 自己决定是否检索、是否可信、是否引用。
- **CRAG (Corrective RAG)** (Yan et al., 2024, arXiv:2401.15884)：检索质量打分 + 检索失败时触发网络/备选；**离线评测不能联网，但"检索失败 → 改用 BM25 兜底"的结构可以保留**。
- **HyDE** (Gao et al., 2022, arXiv:2212.10496)：先让 LLM 生成一段假设性回答再检索，提升长查询-短文档召回；代价是多 1 次 LLM 调用。
- **GraphRAG** (Edge et al., 2024, arXiv:2404.16130)：把检索对象从"chunk"升级到"实体-关系图"。**在本赛长文档 128K+ 场景下是强候选方案，但构图本身要消耗 LLM 预算，要做成离线预计算**。
- **LongRAG** (Jiang et al., 2024, arXiv:2406.15319 *待核实*)：用更长的 chunk（4K+）配合长上下文 LLM，降低检索步骤数。**Qwen 系列长上下文支持较好，这条路线值得尝试**。

### 4.3 轻量 embedding / reranker（CPU 友好）

- **bge-small-en / bge-m3** (Xiao et al., 2023, arXiv:2309.07597 *待核实*)：中英双语，多粒度检索，**官方 Hugging Face 权重在 CPU 上 ~ 数 ms/query，适合本赛**。
- **GTE / E5 小模型** (Wang et al., 2022, arXiv:2212.03533 *待核实*)：微软开源 embedding，5.5B/base/small 多档。
- **bge-reranker-base / jina-reranker-v2**：cross-encoder 重排，CPU 推理可接受（几十毫秒/对）。

> **本赛建议**：**bge-m3 + bge-reranker-base + BM25** 三者级联；对 128K 长文档用 **LongRAG 风格的粗粒度 chunk (~4K)**，然后由 Agent 决定是否展开到细粒度。GraphRAG 留给 Phase 2 的提升空间。

## 5. 多模态数据（Phase 2 图像/视频）

> **重要更正**：Qwen3.5-35B-A3B **本身就是多模态模型**（HF 模型卡标签 Image-Text-to-Text，声称同时支持视频理解；MMMU 81.4 / VideoMME 86.6）。这意味着 **图像/视频可以直接由评测模型端到端处理**，不必再外挂独立 VLM。本节保留作为备选/对比参考。

### 5.1 通用 VLM（作为对照与数据增强手段）

- **LLaVA / LLaVA-1.5 / LLaVA-NeXT** (Liu et al., 2023, arXiv:2304.08485 与后续)：开源多模态 SFT 的主线。
- **Qwen-VL / Qwen2-VL / Qwen2.5-VL** (Bai et al., 2023, arXiv:2308.12966 起)：与本赛主模型同源的前代 VLM，可用于本地开发期对比评估。
- **InternVL2 / InternVL2.5** (Chen et al., 2024, arXiv:2404.16821 起)：开源 VLM SOTA 之一。

### 5.2 图表/数据图像理解（Phase 2 重点）

- **ChartQA** (Masry et al., 2022, arXiv:2203.10244)：图表问答基准。
- **ChartLLaMA / ChartInstruct / ChartAssistant**（2023–2024）：在 ChartQA 等基准上做 LLaMA 适配。
- **Deplot** (Liu et al., 2023, arXiv:2212.10505)：把 chart 先 OCR 成 table，再用 LLM 解答；**"先结构化再推理"的思路仍值得采纳**——可以让 A3B 先用 prompt 把图表转为中间 CSV，再在此基础上做聚合查询，避免单次调用同时承担"看图"和"推理"。
- **MatCha** (Liu et al., 2022, arXiv:2212.09662)：图-文对齐预训练。

### 5.3 视频理解

- **Video-LLaVA / VideoChat / LLaMA-VID**：长视频帧抽取 + token 压缩。
- A3B 官方已具备视频理解（VideoMME 86.6），但视频 token 占用极高；Phase 2 视频任务仍建议：
  1. **离线关键帧抽取**（`ffmpeg` + 简单场景切分），把 N 秒视频抽到 10–30 帧；
  2. 只把关键帧 + 字幕/音频转写文本 交给 A3B；
  3. 对长视频用"窗口 + 摘要"两段式，防止爆上下文。

> **本赛建议（更新后）**：
> - **默认走 A3B 端到端**：图像/视频直接随任务 prompt 传给评测 LLM。
> - **Deplot 式"图→表"管线** 作为降本备选：当 A3B 的多模态调用 token 爆炸时（256K 上下文但输出只要列值），先让一次"图→CSV"调用把图结构化，再让后续 SQL 步骤省掉反复看图。
> - **严格禁用外部 VLM 作为评测阶段主干**（违反赛规"禁止以非 Qwen LLM 作为主推理引擎"）；仅限本地开发期做对比。

## 6. 端到端 Data Agent 系统调研

这一节是赛题最直接的参考工作。

- **Data-Copilot** (Zhang et al., 2023, arXiv:2306.07209)：金融数据分析 Agent，设计了"接口层 - 规划层 - 执行层"三段式架构，**在接口数量 > 100 时仍能稳定运行**。对本赛"多工具调度"的架构直接可用。
- **DataLab** (Weng et al., 2024, arXiv:2412.02205 *待核实*)：面向探索性数据分析（EDA）的 Agent，强调交互与可视化——**Creative 子赛道的最佳参考之一**。
- **InsightPilot** (Ma et al., 2023, arXiv:2304.00477 *待核实*)：自动洞察生成，Agent 主动提出分析问题。
- **ChatDB** (Hu et al., 2023, arXiv:2306.03901)：把 SQL DB 作为 LLM 的"外部记忆"。架构与本赛一致：LLM + SQL 执行器。
- **DS-Agent** (Guo et al., 2024, arXiv:2402.17453 *待核实*)：面向数据科学代码生成的 Agent，引入 case-based reasoning。
- **LLM-Symbolic Programmer (LLM-SP) / Program-of-Thoughts** (Chen et al., 2022, arXiv:2211.12588)：**让 LLM 写 Python 代码、由解释器算答案**，在数值推理上显著优于 CoT。
- **ReAct + Code Interpreter / OpenInterpreter**：工业界普遍做法。在本赛 Docker 环境里用 Python subprocess 执行 pandas/duckdb 是最可靠的数值计算通路。

### 6.1 数据分析 Agent 的相关基准

- **DA-Bench / DA-Code** (Hu et al., 2024, arXiv:2410.07331 *待核实*)：数据分析代码生成评测集。
- **InfiAgent-DABench** (Hu et al., 2024, arXiv:2401.05507 *待核实*)：面向数据分析 Agent 的系统性评测。
- **TableBench** (Wu et al., 2024, arXiv:2408.09174 *待核实*)：表格数据分析综合 benchmark。
- 这几个基准能直接作为 **本地自建评测集的补充来源**，特别是 Starter Kit 演示数据太少时。

## 7. 评测指标与本地评测集建设

### 7.1 本赛官方评分复现

`Score = Recall − λ · (Extra Columns / Predicted Columns)`，其中列以 **multiset 比对**，忽略列名、忽略行序。实现要点：

1. **数值归一化**：统一保留 2 位小数、日期转 ISO 8601、空值转空串。
2. **列匹配算法**：对每条预测列，穷举与金标准列计算 multiset 相等；用匈牙利匹配 (scipy.optimize.linear_sum_assignment) 做 one-to-one 最优对齐。
3. **稳定性**：引入浮点 tolerance（如 `abs(a-b) < 1e-6`），避免评分因 float 抖动掉分。

S2 必须在第一周把评分器写好，且在所有 demo tasks 上本地分数与 Starter Kit 对齐。

### 7.2 本地评测集

- 把官方 demo（Easy/Medium/Hard/Extreme 四档）**按难度分层抽样**作为 dev/held-out。
- 补充来源（上一节提到）：BIRD、DA-Code、InfiAgent-DABench 的子集，改写成本赛 prediction.csv 格式。
- 建议 **每个改动都对全评测集跑一遍**，不要只跑 demo；否则 30 次的线上提交很快会被浪费掉。

### 7.3 不确定性与预算控制

- 每个 task 的 LLM 调用次数要有 **hard cap**（例如默认 12 次），超过则用最后一个候选兜底。
- 用 **早停 + self-consistency**（Wang et al., 2022, arXiv:2203.11171）：对 Hard/Extreme 任务跑 3–5 次采样，按列签名一致性投票。
- 整个 pipeline 要有 **总时长 watchdog**：单 task 超时则跳过并记录一个最合理的空表，避免整包 12 h 超时归零。

## 8. 针对本赛题的技术方案建议（落地层）

### 8.1 推荐架构（Phase 1）

```
┌──────────────────── Orchestrator (Plan-and-Execute) ────────────────────┐
│                                                                         │
│   [1] Task Router           判定任务形态（CSV/JSON / DB / 长文档 / 混合）│
│         │                                                               │
│         ▼                                                               │
│   [2] Context Builder       schema linking + value retrieval            │
│                             长文档分块 + bge-m3 embedding + rerank      │
│         │                                                               │
│         ▼                                                               │
│   [3] Plan Generator (LLM)  生成 JSON 计划（step_type, inputs, outputs) │
│         │                                                               │
│         ▼                                                               │
│   [4] Python Sandbox        pandas / numpy 为主 · duckdb/sqlite3 按需   │
│        + doc_to_table       + Retriever（bge-m3） · LLM IE 工具         │
│         │                                                               │
│         ▼                                                               │
│   [5] Answer Composer (LLM) 把中间表组装成最终 prediction.csv           │
│         │                                                               │
│         ▼                                                               │
│   [6] Self-Verifier (LLM)   列数、类型、值域自检，失败触发重规划         │
│                                                                         │
└─────────────────────────────────────────────────────────────────────────┘
```

- **LLM**：只用 Qwen3.5-35B-A3B（256K 上下文 + 原生多模态），通过 function calling + JSON schema 接口；注意 thinking 模式会吃输出 token，建议在部分子任务（如简单值查询）显式关闭。
- **Embedding**：bge-m3 (CPU, onnx 量化)。
- **Rerank**：bge-reranker-base (CPU)。
- **执行层统一为 Python 沙箱**（Program-of-Thoughts 范式，arXiv:2211.12588）：
  - LLM 产出 Python 代码，由白名单沙箱执行，环境含 `pandas / numpy / duckdb / sqlite3`。
  - **只有当数据源是 `context/db/*.db`（Medium 的主形态）时**才使用 SQL——通过 `duckdb.sql(...)` 或 `pd.read_sql(...)` 调入，避免把大表 (~百 MB) 整库加载进内存。
  - Easy 小 CSV/JSON、Hard/Extreme 的 IE 产物（mini-table，通常 < 100 行）**一律 pandas 直接处理**，不再走 SQL 引擎（数据规模不值得起 DB、且 pandas 对"取 corrected 值/忽略 NaN"等定制逻辑更自然）。
- **IE 子工具 `doc_to_table`**：给定 `doc/*.md` + question + 目标 schema，调 LLM 把散文抽成结构化 `DataFrame`，带 per-cell confidence。产物交给 Python 沙箱下一步聚合。
- **图像/视频**：直接传入 A3B 的多模态接口；若某任务图像多、反复调用成本高，降级为"先一次性转 CSV / 文本摘要，后续步骤只用文本"。
- **缓存**：LRU on-disk cache (diskcache) 缓存 embedding 与 LLM 响应，开发期省钱（API 价档极低也仍有意义）。

### 8.2 Phase 2 升级点（按优先级）

1. **多模态端到端调用的 prompt 设计**：Phase 2 图像/视频直接走 A3B，重点是压缩输入（下采样、关键帧）与严格 JSON 输出。
2. **GraphRAG 预构图**（对知识文档密集型任务）。
3. **多采样 + self-consistency 投票**（攻 Extreme 难度）。
4. **规划 Agent / 执行 Agent 分离**（Creative 子赛道拿分点）。

### 8.3 技术风险与对策

| 风险 | 对策 |
|------|------|
| Qwen3.5-35B-A3B MoE 吞吐/延迟（非显存问题） | 严格限制单 task LLM 调用次数、多步合并为一次调用、缓存中间步骤；必要时关闭 thinking 模式 |
| thinking 模式输出 token 失控 | 对简单子任务传 `enable_thinking=False`；设置 `max_output_tokens` 硬上限 |
| 长文档/长图像流超上下文 | 优先 RAG 召回 top-k chunk；图像做下采样；视频做关键帧抽取 |
| CPU embedding 吞吐低 | 离线预计算：Docker build 时就把 demo 的 embedding 写进镜像 |
| 评测列签名严格 | S2 本地评测器必须 1:1 复现；所有数值/日期走统一归一化 |
| 30 次提交预算用完 | 本地评测集代替排行榜迭代；线上只用于里程碑验证 |
| 开发期 LLM 调用开销 | 全程走阿里云百炼 API，见 §附录 A 成本估算 |

### 8.4 Creative 子赛道的加分点

1. **透明决策链**：把 Plan → Execute 的每一步（包含输入、SQL、结果摘要）写成可视化时间线。
2. **交互式 Agent UI**：用 Gradio / Streamlit 做一个展示用 demo（不用于评测），展示 replay 能力。
3. **错误溯源**：当答案错误时，能从最终列逆推到导致错误的中间步骤。参考 [DataLab] 的交互范式。
4. **文档**：写一份 **Technical Report**，定位为"可投 KDD/SIGMOD Demo Track"的级别。

## 附录 B：Phase 1 Demo 数据观察笔记（2026-04-20 实测）

实测脚本见 [`scripts/profile_demo.py`](../scripts/profile_demo.py)，数据位于 `data/demo/public/{input,output}/`（461 MB zip，解压 1.8 GB，不入版本库）。

### B.1 规模与难度分布

| 项 | 值 |
|----|----|
| 总 task 数 | **50** |
| Task ID 范围 | `task_11` ~ `task_420`（非连续，官方池估计 420+） |
| 难度分布 | easy 15 / medium 23 / hard 10 / **extreme 仅 2** |
| 解压总大小 | 1.8 GB（单 task 最大 442 MB，最小 28 KB） |

**含义**：demo 的 extreme 样本量 = 2，无法直接用来做统计显著的分数估计，**必须从外部基准扩增 hard/extreme 类别**（§B.5）。

### B.2 Task 目录结构（来自 `data/public/README.md`）

```text
task_<id>/
├── task.json       { task_id, difficulty, question }
└── context/
    ├── csv/        *.csv              — 结构化表格
    ├── db/         *.db (SQLite)      — 关系数据库
    ├── json/       *.json             — 结构化 JSON
    ├── doc/        *.md               — 叙事型长文档
    └── knowledge.md                   — 数据库语义说明
```

各目录均为**可选**；每个 task 至少有一种。`knowledge.md` 几乎 100% 出现。

### B.3 数据来源：BIRD 改写

demo 中可确认的源数据库（均来自 BIRD, arXiv:2305.03111）：

- `thrombosis_prediction`（患者/检验/检查，医学类）
- `superhero`（漫威超能力）
- `codebase_community`（StackExchange 问答）
- `european_football`（欧洲足球比赛）
- `card_games`（Magic: The Gathering）

**铁证**：`gold.csv` 的列名直接是 SQL alias：

```sql
COUNT(DISTINCT T1.ID)                        -- task_344
SUM(T3.cost)                                 -- task_163
CAST(SUM(CASE WHEN ... AS REAL) * 100 / COUNT(T1.id)   -- task_420
```

**推论**：

1. BIRD 的剩余样本 + Spider/Spider 2.0 均可**低成本改写为 held-out**（只要执行原 SQL 取 DataFrame，列名保留 alias）。
2. 评分器**必须忽略列名**，只比对列值 multiset——与规则页 §6.5 一致。
3. 选手 prompt **不应要求 LLM 起有语义的列名**。

### B.4 难度的真实含义（按数据模态升维，不是 SQL 复杂度）

| 难度 | 新增模态 | 核心挑战 | 典型 knowledge.md / doc 规模 |
|------|----------|----------|------------------------------|
| easy | CSV / JSON | Python 统计 + 短 knowledge.md | knowledge.md ~6 K chars |
| medium | + SQLite DB | NL2SQL + 多源联合 | 同上 |
| hard | + `doc/` 叙事文档 | 从散文中抽数值再聚合 | doc 10–180 K chars |
| extreme | `doc/` 更长 | 超长文档记忆与 context 工程 | doc ≥ 280 K chars，task_420 doc 1.1 MB |

### B.5 Hard / Extreme 的真实难点（来自人工查样）

以 `task_418` (extreme) 的 `doc/Laboratory.md` 为例，**结构化表被故意改写成叙事段**：

> "The subject's GOT was **initially logged as 35.0 U/L**; however, a calibration review led to a **corrected value of 28.0 U/L**."

三类陷阱：

1. **初始值 vs 修正值并列出现**：正确答案永远是"corrected/final/revised"那一个。
2. **噪声混入**：夹杂"图书借阅记录"、"食堂消费偏好"、"参加体育俱乐部"等与 question 完全无关的生活琐事。
3. **跨段落聚合**：每位患者一段，需抽 N 段后做行级聚合（count、avg、比值）。

**这不是纯 RAG 任务**——chunk 切细会丢值，切粗会稀释召回。正确路径是 **IE（信息抽取）→ 中间表（DataFrame）→ Python/pandas 聚合** 的三段式。

> **为什么不是 SQL**：抽出来的 mini-table 通常只有十几到几十行，整张表直接进 pandas；且"取 corrected 值优先于 initial 值"、"忽略 NaN 再算比值"这类逻辑用 pandas 比用 SQL CASE WHEN + 子查询干净得多。只有当数据源本身就是 `context/db/*.db`（Medium 的形态）时 SQL 才有优势——省内存、schema 现成。

本队 §8 架构图里 **Executor Pool 必须内置一个 `doc_to_table` IE 工具**，以 LLM + 结构化 schema prompt 驱动，产物交给下游 Python 代码步骤。

### B.6 `knowledge.md` 是官方给的"业务语义字典"

示例（`task_344/context/knowledge.md` 节选）：

```text
- Inpatient vs. Outpatient Ratio for Males:
  Formula: COUNT(ID) where Admission='+' and SEX='M' /
           COUNT(ID) where Admission='-' and SEX='M'
```

几乎每个 task 的 `knowledge.md` 都带：表名、列名解释、业务 KPI 公式。必须作为 **schema linking 的最高优先输入**灌入 prompt，而不是只靠列名推理。

### B.7 对方案的直接修正项（覆盖 §5、§7、§8）

1. **§5 多模态**：保持原结论（A3B 端到端），但 Phase 1 里**根本没有图像/视频**——图像类是 Phase 2 才引入，当前全部精力投结构化 + 长文档。
2. **§7.2 held-out 构建**：优先级变为 **BIRD 改写 > 自制长文档 IE task > Spider**，因为 demo 直接就是 BIRD subset。
3. **§8.1 架构（已更新）**：
    - Executor 层 **不做 "SQL endpoint" 抽象**——统一为 Python 代码沙箱（PoT 风格）。
    - SQL 作为沙箱里的**工具调用**（`duckdb.sql / pd.read_sql`），仅在"数据源是 `context/db/*.db`"时使用。
    - 新增 `doc_to_table` 工具负责 `doc/*.md` → mini-table IE；产物直接由 pandas 在 Python 里聚合，不经过 SQL。
4. **§8.2 Phase 2 升级项**：`extreme 仅 2 样本`→ Phase 1 末要对 hard/extreme 做专项 over-sampling 评估，避免方案在难档位上分数方差过大不自知。
